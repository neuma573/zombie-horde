# 최소 플레이어블 버전 설계

## 문서 목적과 현재 저장소 상태

이 문서는 `AGENTS.md`를 기준으로 브라우저에서 실행되는 2D 탑다운 좀비 생존 게임의 최소 플레이어블 버전(MVP)을 정의한다. 현재 저장소에는 게임 코드, `package.json`, 테스트 및 빌드 설정이 없으므로 신규 프로젝트를 시작하는 상태를 전제로 한다.

설계의 핵심 경계는 다음과 같다.

1. 무기가 발사 가능 여부를 결정한다.
2. 순수 hitscan 로직이 즉시 명중 대상을 결정한다.
3. Damage 시스템이 판정 결과를 체력에 반영한다.
4. 시각 효과는 확정된 판정 결과만 소비한다.

기본 총기는 이동하는 `Bullet` 객체를 생성하지 않는다. 실제로 이동하고 충돌하는 `Projectile`은 로켓 런처 같은 특수 무기를 구현하게 될 때 별도 경로로 추가한다.

## 1. MVP 포함 범위

- Phaser 3와 TypeScript로 실행되는 단일 브라우저 게임 화면
- 임시 도형으로 표시되는 캐릭터 1종
- 키보드 이동과 마우스 조준
- 기본 hitscan 무기 1종
- 무기의 공격력, 사거리, 발사 간격, 탄창, 예비 탄약, 재장전 시간, 최대 명중 대상 수
- 임시 도형으로 표시되는 좀비 1종
- 플레이어를 향한 좀비의 단순 직선 이동
- 좀비의 접촉 범위 공격과 공격 간격
- 플레이어와 좀비의 체력, 피해, 사망
- 웨이브 단위로 좀비를 생성하는 무한 방어 모드
- 플레이어 사망 후 Game Over와 현재 게임 재시작
- 총구 화염, 탄도선, 피격 표시의 최소 시각 효과
- 이동, 발사, 재장전, 좀비 공격, 생성 간격이 프레임 속도와 독립적으로 동작하는 로직
- Phaser에 의존하지 않는 핵심 게임 판정의 단위 테스트
- `npm test`와 `npm run build`로 검증 가능한 프로젝트 구성

MVP 입력 방식은 별도 결정이 없다면 WASD 이동, 마우스 위치 조준, 마우스 왼쪽 버튼 발사, 키보드 재장전으로 시작한다.

## 2. MVP 제외 범위

- 스토리 모드
- 남성/여성 캐릭터 선택 및 복수 외형
- 캐릭터별 능력치
- 무기 구매, 습득, 교체 및 인벤토리
- 복수 무기
- 로켓, 수류탄 등 실제 Projectile
- 이동하는 Bullet 객체
- 단발, 볼트액션, 펌프액션을 모두 포괄하는 범용 무기 상태 머신
- 스프라이트, 정식 애니메이션, 사운드 및 완성 에셋
- 경험치, 레벨업, 스킬, 아이템 드롭
- 맵 장애물, 엄폐, 길 찾기 및 복잡한 충돌
- 저장, 계정, 백엔드 및 온라인 기능
- 모바일과 게임패드 입력
- 복잡한 메뉴, 설정 화면 및 일시정지 화면
- 측정으로 필요성이 확인되지 않은 객체 풀링과 공간 분할

## 4. 최소 클래스와 모듈 및 책임

### `GameScene`

- Phaser 객체의 생성과 제거
- 키보드 및 마우스 입력 수집
- Phaser가 전달한 `deltaMs`를 각 시스템에 전달
- 엔티티 상태를 순수 로직 입력 데이터로 변환
- 발사, hitscan, 피해, 효과를 순서대로 연결
- 웨이브 상태와 Game Over 상태 연결
- 최소 HUD 텍스트 갱신

게임 규칙의 계산식은 직접 소유하지 않는다.

### `Player`

- 플레이어 임시 도형과 월드 위치
- 현재 체력과 생존 상태
- 현재 또는 마지막 유효 조준 방향
- 순수 이동 계산 결과를 Phaser 위치와 회전에 반영

무기 탄약과 재장전 상태는 소유하지 않는다.

### `Zombie`

- 좀비 임시 도형과 월드 위치
- 현재 체력과 생존 상태
- hitscan 판정용 원형 충돌 반경
- 순수 추적 이동 결과를 Phaser 위치에 반영

웨이브와 생성 정책은 알지 못한다.

### `WeaponSystem`

- 현재 탄창과 예비 탄약
- 발사 간격과 다음 발사 가능 여부
- 재장전 진행 및 완료
- 발사 요청의 성공 또는 실패 결정
- 성공한 발사에만 탄약 소비
- 성공 시 hitscan에 필요한 무기 설정 반환

대상 탐색, 명중 판정, 피해 적용 및 효과 생성은 하지 않는다.

### `HitscanSystem`

- 살아 있는 좀비에서 ID, 위치, 반경만 수집
- Phaser 비의존 `resolveHitscan` 순수 함수 호출
- 가까운 순서의 명중 결과와 탄도선 종료점 반환

피해를 적용하거나 효과를 그리지 않는다.

### `DamageSystem`

- 확정된 명중 대상에 피해 적용
- 체력이 0 이하인 대상의 사망 확정
- Scene이 제거와 효과 처리에 사용할 결과 반환

ray 판정이나 시각 효과를 담당하지 않는다.

### `SpawnSystem`

- 현재 웨이브에서 남은 생성 수 관리
- `deltaMs` 기반 생성 간격 처리
- 화면 가장자리의 유효 생성 위치 선택
- Scene에 좀비 생성 요청 반환

### `WaveSystem`

- 현재 웨이브 번호와 단계 관리
- 웨이브별 총 생성 수 계산
- 생성 완료 여부와 생존 좀비 수를 바탕으로 다음 단계 결정
- 다음 웨이브 대기 시간 처리

좀비 인스턴스를 직접 소유하지 않는다.

### 효과 모듈

- `MuzzleFlashEffect`: 발사 성공 위치에 짧은 총구 화염 표시
- `TracerEffect`: 확정된 ray 원점과 종료점 사이의 탄도선 표시
- `HitEffect`: 확정된 명중점에 짧은 피격 표시

효과는 명중 여부나 피해 결과를 변경하지 않는다. 효과 생성이 실패하거나 생략되어도 게임 판정은 동일해야 한다.

### `logic` 모듈

상태를 계산하는 Phaser 비의존 함수 모음이다. Phaser 객체, Scene, Graphics 및 Arcade Physics 타입을 입력으로 받지 않는다. 입력 데이터와 계산 결과만 주고받는다.

## 5. Hitscan 명중 판정과 관통 규칙

### 판정 과정

1. 입력이 들어오면 `WeaponSystem`이 탄약, 발사 간격 및 재장전 상태를 검사한다.
2. 실제 발사에 성공했을 때만 hitscan 판정을 시작한다.
3. 플레이어 또는 총구의 월드 좌표를 발사 원점 `origin`으로 정한다.
4. 원점에서 마우스 월드 좌표를 향하는 방향 벡터를 계산하고 정규화한다.
5. 방향 벡터 길이가 0에 가까우면 마지막 유효 조준 방향을 사용한다.
6. 살아 있는 좀비만 후보로 수집한다.
7. 각 좀비를 원으로 취급하여 ray와 원의 최초 교차점을 계산한다.
8. 원점 뒤에 있거나 무기 사거리 밖인 교차점은 제외한다.
9. 최초 교차 거리 `t`의 오름차순으로 후보를 정렬한다.
10. 같은 거리에서는 고유한 좀비 ID를 보조 키로 사용해 결과를 결정적으로 만든다.
11. 무기의 최대 명중 대상 수만큼 가까운 순서로 선택한다.
12. 선택된 대상과 명중점만 Damage 시스템에 전달한다.
13. 피해와 사망을 확정한 뒤 결과를 시각 효과에 전달한다.

추천하는 순수 로직의 개념적 입출력은 다음과 같다.

```ts
resolveHitscan(
  origin,
  direction,
  range,
  targets,
  maxTargets,
): {
  hits: Array<{ targetId; distance; point }>;
  endPoint;
}
```

### 비관통 규칙

- 내부 설정에서 `maxTargets = 1`로 표현한다.
- ray와 교차하는 가장 가까운 대상 하나만 명중한다.
- 대상의 중심점이 아니라 ray가 대상 원에 최초 진입하는 거리로 가까움을 판단한다.
- 첫 대상의 교차점에서 판정을 종료한다.
- 탄도선도 첫 대상의 교차점까지만 표시한다.
- 뒤의 대상은 같은 ray 위에 있어도 피해를 받지 않는다.

### 관통 규칙

- `maxTargets > 1`로 표현한다.
- 교차 후보를 최초 교차 거리순으로 정렬하고 가까운 대상부터 최대 `maxTargets`개를 명중시킨다.
- 후보가 설정 수보다 적으면 존재하는 대상만 명중한다.
- MVP에서는 거리나 관통 순서에 따른 피해 감소를 적용하지 않는다.
- 명중 수가 `maxTargets`에 도달하면 마지막 대상의 최초 교차점에서 ray 판정을 종료한다.
- 명중 수가 `maxTargets`에 도달하지 않으면 최대 사거리에서 종료한다.
- 탄도선은 판정에서 계산된 동일한 종료점을 사용한다.

`AGENTS.md`의 `관통 수`는 전체 명중 대상 수인지 첫 대상 이후 추가 관통 수인지 모호하다. 구현에서는 우선 의미가 분명한 `maxTargets`, 즉 “한 발이 명중시킬 수 있는 최대 대상 수”를 사용할 것을 권장한다. 외부 설정명을 반드시 `penetrationCount`로 해야 한다면 구현 전에 의미를 확정한다.

## 6. 게임 루프와 상태 흐름

### 상위 상태

```text
Playing → GameOver → Playing(재시작)
```

MVP에서는 별도 시작 메뉴 없이 게임을 바로 시작할 수 있다.

### 웨이브 상태

```text
Waiting → Spawning → Active → Waiting
```

- `Waiting`: 다음 웨이브 시작까지 대기
- `Spawning`: 설정된 간격으로 이번 웨이브 좀비 생성
- `Active`: 생성은 끝났지만 살아 있는 좀비가 남아 있음
- 생성이 끝나고 생존 좀비가 0명이면 다음 `Waiting`으로 전환

### 한 프레임의 권장 처리 순서

1. Phaser에서 `deltaMs` 수신
2. Game Over이면 재시작 입력만 처리
3. 플레이어 이동 입력을 순수 이동 함수에 전달
4. 이동 결과를 월드 경계 안으로 제한하여 반영
5. 마우스 월드 좌표로 조준 방향 갱신
6. 무기의 발사 및 재장전 타이머 갱신
7. 발사 성공 시 hitscan 판정
8. 명중 대상의 피해와 사망 처리
9. 죽은 좀비를 생존 목록에서 제거
10. 살아남은 좀비의 추적 이동 계산 및 반영
11. 좀비 접촉 공격 간격과 플레이어 피해 처리
12. 플레이어 사망 여부 확인
13. Spawn 및 Wave 타이머 갱신
14. 생성 요청에 따라 좀비 생성
15. 효과 수명과 HUD 갱신

같은 프레임에 플레이어가 좀비를 죽인 경우 해당 좀비가 다시 이동하거나 공격하지 않도록 사격과 사망 처리를 먼저 수행한다.

### 프레임 속도 독립성

- 이동은 `position += velocity * deltaSeconds`로 계산한다.
- 발사, 재장전, 접촉 공격 및 생성은 누적 밀리초나 게임 시각을 기준으로 처리한다.
- 반복 타이머는 큰 delta에서도 이벤트를 잃지 않도록 누적 시간을 간격 단위로 소비한다.
- 브라우저 탭 복귀 시 지나치게 큰 delta가 한 프레임에 적용되지 않도록 gameplay delta 상한을 설정하는 방안을 사용한다.

delta 상한은 긴 정지 시간 일부를 버리는 정책이므로 구현 전에 상한값과 적용 대상 타이머를 확정한다. MVP에서는 별도 고정 timestep 시뮬레이션 엔진을 선행 구축하지 않는다.

## 7. 테스트할 순수 로직

### 이동

- 같은 총 시간을 서로 다른 delta 분할로 처리해도 같은 최종 위치가 되는지
- 대각선 이동이 축 방향 이동보다 빠르지 않은지
- 좀비가 플레이어 방향으로 속도와 시간에 맞게 이동하는지
- 플레이어가 월드 경계를 벗어나지 않는지

### Hitscan

- ray 위 대상 명중
- ray 밖, 원점 뒤, 사거리 밖 대상 제외
- 비관통 시 가장 가까운 대상 하나만 명중
- 입력 배열 순서와 관계없이 거리순으로 명중
- 관통 시 `maxTargets`까지만 명중
- 후보가 적으면 존재하는 대상만 명중
- 대상 중심이 아닌 최초 ray-circle 교차 거리로 순서 계산
- 동일 거리에서 ID를 사용한 결정적 순서
- 비정규화 방향 입력 처리
- 0에 가까운 방향 입력 처리
- 명중 여부와 최대 명중 수에 따른 ray 종료점

### 무기

- 발사 간격 전 발사 실패
- 발사 가능 시간이 지난 뒤 발사 성공
- 성공한 발사만 탄약 소비
- 빈 탄창에서 발사 실패
- 재장전 중 발사 실패
- 재장전 완료 전 탄약 이동 없음
- 재장전 완료 시 필요한 양만 예비 탄약에서 탄창으로 이동
- 예비 탄약이 부족한 경우의 재장전
- delta 분할이 달라도 같은 쿨다운 및 재장전 결과

### 피해

- 피해량만큼 체력 감소
- 체력이 0 이하가 되면 사망
- 이미 죽은 대상의 중복 사망 이벤트 방지
- 복수 hitscan 대상에 독립적으로 피해 적용

### 웨이브와 생성

- 초기 대기 후 첫 웨이브 시작
- 설정 간격에 맞춰 정확한 수의 좀비 생성
- 생성 완료 후 좀비가 남아 있으면 다음 웨이브로 전환하지 않음
- 생성 완료 및 생존자 0명일 때 다음 대기로 전환
- 웨이브 번호에 따른 수량 증가
- 큰 delta에서도 생성 횟수를 누락하지 않음

Phaser Scene의 통합 테스트는 최소 smoke test만 고려한다. 핵심 판정 테스트는 DOM, canvas 및 Phaser 런타임 없이 실행할 수 있어야 한다.

## 8. 구현 작업과 완료 조건

### 작업 7: 좀비 접촉 공격과 플레이어 피해

#### 범위

- 플레이어 체력 및 생존 상태
- 플레이어와 좀비 간 원형 접촉 판정
- 좀비별 공격 간격
- 접촉 중 플레이어 피해 처리
- 피격 후 짧은 무적 시간
- 플레이어 사망 이벤트 발생
- 접촉 및 피해 계산을 순수 로직으로 분리

#### 구현 규칙

- 좀비는 플레이어와 접촉했다고 매 프레임 피해를 주지 않는다.
- 각 좀비는 독립적인 공격 대기 시간을 가진다.
- 플레이어는 피해를 받은 뒤 설정된 시간 동안 추가 피해를 받지 않는다.
- 플레이어 체력은 0 미만으로 내려가지 않는다.
- 플레이어 사망 처리는 한 번만 발생한다.
- 사망 이후 추가 피해는 무시한다.

#### 완료 조건

- 접촉하지 않은 좀비는 플레이어에게 피해를 주지 않는다.
- 접촉 상태가 유지되어도 공격 간격 이전에는 추가 피해가 발생하지 않는다.
- 여러 좀비가 동시에 접촉했을 때 무적 시간 규칙에 따라 피해가 처리된다.
- 서로 다른 delta 분할에서도 공격 횟수와 피해 결과가 동일하다.
- 큰 delta가 들어와도 공격 대기 시간이 정상적으로 갱신된다.
- 플레이어 체력이 0 이하가 되면 사망 상태로 한 번만 전환된다.
- Phaser 없이 접촉 판정, 공격 간격, 무적 시간 및 사망 테스트가 실행된다.
- `npm test`와 `npm run build`가 통과한다.

#### 제외 범위

- 재시작

- 좀비 넉백

- 회복 아이템

### 작업 8: Game Over와 게임 세션 재시작

#### 범위

- `Playing`, `GameOver` 게임 세션 상태
- 플레이어 사망 시 Game Over 전환
- Game Over 상태에서 이동, 발사, 재장전 및 웨이브 진행 중지
- 재시작 입력 처리
- 플레이어, 무기, 좀비, 웨이브 및 타이머 초기화
- Phaser 이벤트 및 객체 정리

#### 구현 규칙

- 플레이어 사망 이벤트가 발생하면 게임 상태는 한 번만 `GameOver`로 전환된다.
- `GameOver` 상태에서는 게임 도메인 업데이트를 중지한다.
- `GameOver` 상태에서도 화면 렌더링과 재시작 입력 처리는 유지할 수 있다.
- 재시작은 기존 상태를 수정하는 대신 새로운 초기 세션 상태를 생성한다.
- Scene 재시작 또는 세션 재생성 방식 중 하나를 일관되게 사용한다.
- 이벤트 리스너와 타이머가 중복 등록되지 않도록 정리한다.

#### 완료 조건

- 플레이어 체력이 0이 되면 Game Over 상태로 전환된다.
- Game Over 이후 플레이어 이동과 무기 발사가 실행되지 않는다.
- Game Over 이후 좀비 이동, 공격 및 추가 생성이 중지된다.
- 재시작하면 플레이어 체력과 위치가 초기값으로 복구된다.
- 재시작하면 무기 탄창, 예비 탄약, 쿨다운 및 재장전 상태가 초기화된다.
- 재시작하면 모든 좀비가 제거되고 첫 웨이브 대기 상태로 돌아간다.
- 여러 번 재시작해도 이벤트 리스너, 타이머 또는 업데이트 호출이 중복되지 않는다.
- 재시작 이전 세션의 객체가 이후 세션에 영향을 주지 않는다.
- 게임 상태 전환과 초기화 테스트가 통과한다.
- `npm test`와 `npm run build`가 통과한다.

#### 제외 범위

- 메인 메뉴
- 일시정지 메뉴
- 점수 저장
- 최고 기록 저장
- 서버 통신
- 화려한 Game Over 연출

### 작업 9: 최소 반응형 HUD

#### 범위

- 플레이어 체력 표시
- 탄창과 예비 탄약 표시
- 재장전 상태 표시
- 현재 웨이브 표시
- 현재 생존 좀비 수 표시
- 다음 웨이브 대기 상태 표시
- Game Over 및 재시작 안내 표시
- PC와 모바일의 다양한 화면 크기 및 종횡비에 대응하는 최소 반응형 배치

#### 구현 규칙

- HUD는 게임 상태를 변경하지 않는다.
- HUD는 읽기 전용 화면 모델을 받아 렌더링한다.
- 도메인 로직이 Phaser Text 객체를 직접 참조하지 않는다.
- 값이 변경되었을 때만 화면 문자열을 갱신할 수 있도록 한다.
- Scene 재시작 시 HUD 객체가 중복 생성되지 않도록 한다.
- HUD는 특정 고정 해상도나 종횡비를 기준으로 절대 위치를 가정하지 않는다.
- 화면 크기가 변경되면 현재 화면 영역을 기준으로 HUD를 다시 배치한다.
- 모바일 세로 모드를 기본으로 하며 모바일 가로와 PC 화면에서도 같은 필수 정보를 확인할 수 있어야 한다.
- 모바일 브라우저의 safe-area를 고려해 필수 정보가 노치나 시스템 UI에 가려지지 않도록 한다.

#### 완료 조건

- 체력이 변경되면 HUD에 반영된다.
- 발사와 재장전 후 탄약 상태가 정확히 표시된다.
- 웨이브 전환 시 현재 웨이브 번호가 갱신된다.
- 좀비 생성 및 사망 시 생존 수가 갱신된다.
- 재장전 중 상태가 화면에 표시된다.
- Game Over 상태와 재시작 입력 안내가 표시된다.
- 재시작 후 HUD가 초기 상태로 복구된다.
- 여러 번 재시작해도 HUD가 중복 표시되지 않는다.
- 모바일 세로, 모바일 가로 및 대표적인 PC 종횡비에서 필수 HUD가 화면 밖으로 잘리지 않는다.
- 작은 화면에서도 HUD 항목이 서로 겹쳐 읽을 수 없게 되지 않는다.
- 화면 크기 또는 방향이 바뀌면 HUD가 현재 화면 안으로 재배치된다.
- safe-area가 있는 화면에서도 체력, 탄약, 웨이브 및 Game Over 안내를 확인할 수 있다.
- `npm test`와 `npm run build`가 통과한다.

- 짧은 시간에 반복 발사해도 효과 객체가 무한히 누적되지 않는다.
- 효과 종료 후 관련 객체가 제거된다.
- 시각 효과를 비활성화해도 게임 로직과 테스트 결과가 동일하다.
- `npm test`와 `npm run build`가 통과한다.

- 웨이브 대기 시간
- 웨이브별 좀비 수
- 웨이브 증가량
- 좀비 생성 간격
- 플레이어와 좀비 간 최소 생성 거리

#### 구현 규칙

- 핵심 밸런스 숫자를 Scene 또는 엔티티 클래스에 직접 작성하지 않는다.
- 테스트에서는 별도의 설정 값을 주입할 수 있어야 한다.
- 동일한 입력, delta 및 난수 seed를 사용하면 동일한 결과가 나와야 한다.
- 자동 통합 테스트는 Phaser 렌더링 없이 실행할 수 있어야 한다.
- 수동 브라우저 검증 항목은 체크리스트로 관리한다.

#### 완료 조건

- 플레이어가 이동하고 마우스로 조준해 좀비를 공격할 수 있다.
- 탄약이 소모되고 재장전이 정상적으로 동작한다.
- 좀비가 플레이어를 추적하고 접촉 시 피해를 준다.
- 좀비를 모두 처치하면 다음 웨이브가 시작된다.
- 플레이어 사망 시 Game Over 상태가 된다.
- 재시작 후 첫 웨이브부터 다시 정상적으로 플레이할 수 있다.

- 여러 번 재시작해도 이벤트, 타이머 및 객체 수가 비정상적으로 증가하지 않는다.

- 브라우저에서 최소 10분간 플레이했을 때 진행 불가 오류가 발생하지 않는다.
- 콘솔에 처리되지 않은 예외가 출력되지 않는다.
- `npm test`와 `npm run build`가 통과한다.

- 게임 시작 후 별도 설명 없이 이동, 조준, 발사 및 재장전이 가능하다.

- 플레이어와 좀비 모두 피해와 사망 상태를 가진다.
- 웨이브는 종료 횟수 제한 없이 반복된다.
- 플레이어 사망 후 재시작할 수 있다.
- 체력, 탄약, 웨이브 상태를 화면에서 확인할 수 있다.

## 9. 예상 성능 문제

### 발사마다 모든 좀비를 검사하는 비용

단순 구현은 발사마다 모든 생존 좀비에 대해 교차 검사를 수행하고 후보를 정렬한다. 검사 비용은 `O(n)`, 전체 정렬은 `O(n log n)`이다.

MVP 대응:

- 생존 좀비만 후보에 포함한다.
- 사거리와 ray 전방 여부를 먼저 검사한다.
- 동시 생존 좀비 수에 합리적인 밸런스 상한을 둔다.
- 실제 측정에서 병목이 확인되기 전에는 spatial hash나 quadtree를 구현하지 않는다.

### 엔티티와 효과의 반복 생성 및 제거

좀비와 짧은 효과를 자주 생성하고 파괴하면 garbage collection 부담이 생길 수 있다.

MVP 대응:

- 효과 지속 시간을 짧고 일정하게 유지한다.
- 발사당 효과 수를 제한한다.
- 프로파일링에서 문제가 확인되기 전에는 객체 풀을 만들지 않는다.

### 탭 복귀 시 큰 delta

브라우저가 비활성화되었다가 복귀하면 이동, 생성 및 반복 공격이 한 프레임에 몰릴 수 있다.

MVP 대응:

- gameplay delta 상한을 둔다.
- 누적 타이머가 이벤트를 누락하지 않도록 테스트한다.
- 필요하면 한 프레임의 최대 생성 횟수를 제한한다.

### 좀비 겹침

모든 좀비가 같은 플레이어 위치로 직선 이동하면 서로 겹칠 수 있다.

MVP 대응:

- 플레이를 심각하게 방해하지 않으면 허용한다.
- 문제가 확인되면 단순 최소 거리 밀어내기를 별도 작업으로 검토한다.
- 군집 행동이나 길 찾기를 선행 구현하지 않는다.

### 판정과 효과의 결합 위험

렌더링 객체의 수명이나 프레임 지연이 판정에 영향을 주면 게임 결과가 불안정해진다.

MVP 대응:

- hitscan과 피해를 먼저 확정한다.
- 효과는 확정된 원점, 종료점 및 명중점만 입력받는다.
- 효과 모듈은 판정 시스템으로 값을 돌려주지 않는다.

## 11. `AGENTS.md`에서 발견한 충돌이나 모호성

### 명시적인 정면 충돌

현재 문구 사이에 구현이 불가능할 정도의 직접적인 충돌은 없다. 다만 장기 아키텍처 항목과 MVP 범위가 섞여 있어 아래 사항을 임의로 구현하지 않도록 주의해야 한다.

### `Projectile` 디렉터리와 MVP 범위

`Architecture`에는 `Projectile` entity와 system이 제시되지만, MVP는 기본 hitscan 무기만 포함하고 특수 Projectile을 제외한다. 장기 구조 제안으로 해석하되 MVP에서는 빈 Projectile 클래스나 시스템도 만들지 않는다. 특수 무기 구현이 승인될 때 추가한다.

### 관통 수의 의미

`관통 수`가 전체 명중 가능 대상 수인지, 첫 대상 이후 추가로 통과할 수 있는 대상 수인지 정의되지 않았다. 이 문서는 내부 설정 `maxTargets`를 사용하여 전체 최대 명중 대상 수로 표현할 것을 제안한다. 최종 설정 이름과 사용자 표시 의미는 구현 전에 확정해야 한다.

### 기본 무기의 작동 방식

지원 가능한 방식으로 단발, 연사, 볼트액션, 펌프액션이 언급되지만 MVP 기본 무기의 방식은 정해지지 않았다. 가장 작은 구현은 단순 단발 또는 입력 유지 연사 중 하나다. 발사 입력 정책을 구현 전에 결정해야 하며, 선택하지 않은 방식은 만들지 않는다.

### 웨이브 밸런스와 증가 공식

무한 방어 모드는 명시되어 있지만 초기 좀비 수, 웨이브별 증가량, 생성 간격 및 다음 웨이브 대기 시간이 없다. 선형 증가를 기본 제안으로 삼을 수 있으나 구체 값은 config 구현 전에 확정해야 한다.

### 입력 방식

키보드, 마우스, 모바일 또는 게임패드 중 어떤 입력을 사용할지 명시되지 않았다. 브라우저 데스크톱 MVP로 제한하고 WASD와 마우스를 사용하는 것을 제안한다.

### 플레이어 사망 이후 흐름

사망 조건과 재시작 방식이 정의되지 않았다. 최소 플레이 가능성을 위해 Game Over와 즉시 재시작만 추가하고 별도 메뉴는 제외하는 것을 제안한다.

### 프레임 독립성의 구체 정책

프레임 독립 동작은 명시되어 있지만 큰 delta 처리, 고정 timestep 여부 및 delta 상한 정책은 정해지지 않았다. MVP는 delta 기반 계산과 누적 타이머를 사용하고, 탭 복귀 문제를 위한 상한값은 구현 전에 결정한다.

### `한 작업에서 여러 기능`의 작업 단위

기능, 티켓, 커밋 또는 PR 중 무엇을 한 작업으로 보는지 정의되지 않았다. 위험을 피하기 위해 사용자에게 검증 가능한 단일 기능과 테스트를 하나의 구현 작업으로 보고, 웨이브·접촉 공격·Game Over·효과처럼 독립적으로 검증할 수 있는 기능은 분리한다.
