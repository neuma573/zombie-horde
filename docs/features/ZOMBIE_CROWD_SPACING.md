# 좀비 대규모 군중 완화 설계

## 1. 문서 상태

- 이 문서는 좀비가 한 점에 장시간 완전히 포개지는 현상을 줄이는 soft spacing 요구사항을 정의한다.
- 현재 브랜치는 이 문서를 기준으로 순수 군중 완화 로직과 `GameScene` 연결을 구현한다.
- 공간 그리드 후보 정책, 프레임 독립성, 장애물 및 접촉 피해 경계를 유지한다.
- 완전한 좀비–좀비 물리 충돌은 이 문서의 목표가 아니다.

## 2. 작업 목적

- 여러 좀비가 같은 중심에 모여 한 개체처럼 보이는 현상을 완화한다.
- 좀비 떼의 폭과 개체 실루엣을 읽을 수 있게 한다.
- 좀비다운 무질서와 일부 겹침은 유지한다.
- 플레이어 추적을 유지하면서 가까운 좀비 사이에 제한된 반발 이동을 추가한다.
- 기존 공간 그리드 후보 쿼리를 실제 목적에 맞게 처음 소비한다.
- 플레이어 접촉 피해 경로와 플레이어–좀비 충돌 계약을 보존한다.

## 3. 현재 구현 분석

- `GameScene`은 각 좀비를 플레이어 방향으로 `moveToward`한 뒤 장애물 sweep을 적용한다.
- 모든 좀비가 같은 플레이어 좌표를 목표로 하므로 같은 방향과 중심으로 수렴할 수 있다.
- `queryZombieCollisionCandidates`는 가까운 좀비 ID 쌍을 결정론적으로 반환하지만 게임 루프에는 연결되지 않았다.
- 후보 쿼리는 기본 입력 4,096개와 후보 15,000회의 상한을 가지며 `valid`와 `complete`를 반환한다.
- 후보 쿼리의 불완전 결과는 위치 보정에 부분 적용하지 않는 계약이다.
- `separatePlayerFromZombies`는 플레이어–좀비 겹침만 해소하며 좀비–좀비 관계를 처리하지 않는다.
- 접촉 피해는 보정 전 `startPosition → movementEnd` 경로를 사용한다.
- 장애물 이동은 `moveCircleWithObstacles`가 담당한다.

## 4. 범위

### 포함

- 공간 그리드의 완전한 후보 쌍을 이용한 좀비 근접 관계 계산
- 현재 위치 스냅샷을 기준으로 한 대칭적인 반발 속도 계산
- 기존 플레이어 추적 속도와 반발 속도의 합성
- 좀비 속도 상한 유지
- 일부 겹침을 허용하는 목표 최소 거리
- 동일 중심에서의 결정론적 fallback 방향
- 장애물 및 월드 경계를 지키는 최종 이동
- Phaser 비의존 순수 계산 테스트
- `GameScene`의 최소 연결

### 제외

- 좀비 반지름 합만큼 완전히 분리하는 hard collision
- 겹침이 0이 될 때까지 반복하는 위치 projection
- 프레임 간 후보 cursor, 작업 credit 및 backlog
- fixed timestep 또는 별도 물리 시뮬레이션 루프
- 좀비 pathfinding, 벽 우회, flocking 및 formation
- 플레이어–좀비 충돌 로직 변경
- 접촉 피해량, windup, cooldown 및 무적 시간 변경
- 웨이브 수, 생성 간격 및 좀비 속도 밸런스 변경
- 애니메이션, 스프라이트 및 시각 에셋 변경
- 공간 그리드 자체의 알고리즘 변경

## 5. 목표 행동

군중 완화의 기준은 다음 한 문장이다.

> 좀비는 일부 겹칠 수 있지만 여러 개체가 완전히 같은 중심에 장시간 머물지는 않는다.

- 좀비들이 일정 간격의 격자나 대열을 만들 필요는 없다.
- 근접한 좀비는 서로 반대 방향의 약한 속도 보정을 받는다.
- 반발은 플레이어 추적보다 약해야 한다.
- 좀비 한 마리가 다른 좀비 때문에 플레이어 반대 방향으로 도망가면 안 된다.
- 군중 밀도가 높아져도 좀비의 최종 속도가 기본 속도를 넘으면 안 된다.
- 장애물에 막힌 반발 이동은 기존 obstacle sweep에 의해 자연스럽게 제한한다.
- 벽 근처에서 겹침이 남는 것은 허용하며 이를 해결하기 위한 경로 탐색은 추가하지 않는다.

## 6. 설정값

초기 설정은 기존 좀비 반지름 20px와 속도 80px/s를 기준으로 한다.

```ts
interface ZombieCrowdSpacingConfig {
  minimumDistanceRatio: number;
  maximumSeparationSpeed: number;
}
```

- `minimumDistanceRatio`: `0.9`
  - 두 좀비 반지름 합의 90%를 목표 최소 중심 거리로 사용한다.
  - 기본 좀비 두 마리는 36px보다 가까울 때 반발한다.
  - 반지름 합 40px보다 작으므로 약 10%의 시각적 겹침은 허용한다.
- `maximumSeparationSpeed`: `36px/s`
  - 기본 추적 속도 80px/s보다 작다.
  - 반발만으로 플레이어 추적을 압도하지 못하게 한다.

설정은 기존 config 관례에 맞는 한 파일에서 관리하며 Scene에 숫자를 반복하지 않는다. 이번 기능을 위해 범용 steering config나 복수 좀비 유형 시스템을 만들지 않는다.

- `minimumDistanceRatio`는 finite 값만 허용하고 `0..1`로 제한한다.
- `maximumSeparationSpeed`는 finite 값만 허용하고 `0..zombieSpeed` 미만으로 제한한다.
- 비정상 설정은 `NaN`을 전파하지 않고 반발 0인 안전한 설정으로 처리한다.

## 7. 순수 반발 계산

### 입력 스냅샷

반발 계산은 업데이트 시작 시점의 살아 있는 좀비 위치를 소비한다.

```ts
interface CrowdSpacingEntry {
  id: string;
  position: Position;
  radius: number;
}
```

계산 중 Phaser 객체의 위치를 변경하지 않는다.

순수 함수의 결과는 최소한 다음 의미를 제공한다.

```ts
interface CrowdSpacingResult {
  velocities: Map<string, Position>;
  valid: boolean;
}
```

- 입력 ID는 비어 있지 않고 유일해야 한다.
- 후보 쌍의 두 ID는 모두 입력 스냅샷에 존재해야 한다.
- 중복 ID, 알 수 없는 후보 ID 또는 비정상 좌표가 있으면 `valid: false`를 반환한다.
- 실패 결과에는 어떤 좀비에도 부분 반발을 적용하지 않는다.
- `GameScene`은 실패 결과에서 기존 추적 이동만 사용한다.

### 한 후보 쌍의 반발

각 후보 쌍에 대해 다음 값을 계산한다.

```text
minimumDistance = (first.radius + second.radius) * minimumDistanceRatio
penetration = minimumDistance - centerDistance
```

- `penetration <= 0`이면 반발하지 않는다.
- `penetration > 0`이면 두 좀비에 크기가 같고 방향이 반대인 기여도를 추가한다.
- 반발 강도는 `clamp(penetration / minimumDistance, 0, 1)`에 비례한다.
- 반지름 또는 `minimumDistance`가 0이면 해당 쌍은 반발하지 않는다.

### 동일 중심

두 중심이 같으면 현재 좌표만으로 방향을 계산할 수 없다. 이 경우 정규화된 canonical ID 쌍으로 결정론적인 단위 방향을 계산한다.

- 난수, 현재 시각, 렌더 프레임 번호를 사용하지 않는다.
- 같은 ID 쌍에는 항상 같은 방향을 반환한다.
- 두 좀비에는 정확히 반대 방향을 적용한다.

### 다중 이웃

- 각 좀비의 모든 반발 기여도를 합산한다.
- 합산 벡터의 길이를 `maximumSeparationSpeed` 이하로 제한해 군중 밀도만으로 반발 속도가 커지지 않게 한다.
- 계산은 모든 후보를 읽은 뒤 ID별 속도 Map으로 한 번에 반환한다.
- 후보 처리 중간에 좀비 위치를 변경하지 않아 후보 순서가 결과를 바꾸지 않게 한다.

## 8. 추적 속도와 합성

각 좀비에 대해 플레이어 추적 속도와 군중 반발 속도를 합친다.

```text
chaseVelocity = directionToPlayer * zombieSpeed
combinedVelocity = chaseVelocity + separationVelocity
finalVelocity = clampMagnitude(combinedVelocity, zombieSpeed)
```

- 추적 속도는 기존 `ZOMBIE_CONFIG.speed`를 사용한다.
- 최종 속도는 기존 좀비 속도를 넘지 않는다.
- `maximumSeparationSpeed < zombieSpeed`를 유지하므로 반발만으로 정상적인 추적 방향을 완전히 뒤집지 않는다.
- 플레이어와 같은 위치라 chase 방향이 0이면 추적 이동을 종료한다. 이후의 실제 겹침은 기존 플레이어–좀비 분리 단계가 처리한다.
- 반발 후보가 없으면 기존 추적 이동 결과와 같아야 한다.

최종 이동은 다음과 같다.

```text
desiredEnd = startPosition + finalVelocity * deltaMs / 1000
resolvedEnd = moveCircleWithObstacles(startPosition, desiredEnd, ...)
```

플레이어 방향 이동 성분이 남은 거리에 도달하면 횡방향 반발을 포함한 해당 이동 전체를 플레이어 위치에서 종료한다. 플레이어 방향의 초과분만 제거하고 횡방향 이동을 남기면 큰 delta 한 번과 분할 delta의 종점이 달라지므로 허용하지 않는다.

## 9. 공간 그리드 결과 정책

- 살아 있고 활성 상태인 좀비만 후보 쿼리에 전달한다.
- `valid: true`, `complete: true`인 결과만 군중 완화에 사용한다.
- `valid: false`이면 해당 업데이트의 군중 반발을 전부 생략한다.
- `complete: false`이면 부분 후보를 사용하지 않고 해당 업데이트의 군중 반발을 전부 생략한다.
- 실패하거나 불완전해도 기존 플레이어 추적 이동은 계속한다.
- 일부 ID에만 반발을 적용하는 fallback은 구현하지 않는다.
- 후보 상한을 delta에 따라 변경하지 않는다.
- 프레임 간 후보 cursor나 backlog를 만들지 않는다.

이 정책은 후보 순서에 따른 일부 좀비 편향과 FPS별 부분 적용 차이를 방지한다.

## 10. 게임 루프 순서

관련 업데이트 순서는 다음과 같다.

```text
1. playerStart 저장
2. 플레이어 이동 및 playerMovementEnd 저장
3. zombieStarts 저장
4. zombieStarts 스냅샷으로 공간 그리드 후보 쿼리
5. 완전한 후보 결과로 ID별 separationVelocity 계산
6. 각 좀비의 chaseVelocity와 separationVelocity 합성
7. deltaMs로 desiredEnd 계산
8. 장애물·월드 경계를 적용해 zombieMovementEnds 저장
9. 기존 startPosition → movementEnd 경로로 접촉 피해 계산
10. 기존 플레이어–좀비 겹침 해소
11. resolvedPosition을 Phaser 객체와 카메라에 반영
12. 기존 회전·HUD·Effect·Game Over·웨이브 처리
```

- 군중 완화 이동은 좀비의 실제 이동 경로에 포함되므로 `zombieMovementEnd`에 반영한다.
- 접촉 피해는 군중 완화와 장애물 이동을 적용한 실제 경로를 사용한다.
- 플레이어–좀비 충돌 보정 후 위치를 접촉 endpoint로 사용하지 않는 기존 계약을 유지한다.
- 새로 생성된 좀비는 다음 업데이트의 스냅샷부터 군중 완화에 참여한다.
- Game Over에서는 기존 조기 반환으로 군중 완화를 실행하지 않는다.

## 11. delta 및 프레임 독립성 정책

- 반발 결과는 px/frame이 아니라 px/s 속도다.
- 이동 거리는 `velocity * deltaMs / 1000`으로 계산한다.
- 후보 수나 이웃 수에 따라 delta를 변경하지 않는다.
- 큰 delta에서도 경과 시간을 버리거나 별도 작업 부채를 만들지 않는다.
- 후보 탐색과 반발 속도 계산은 `deltaMs`를 받지 않는다.
- 같은 위치 스냅샷과 목표에는 FPS와 무관하게 같은 속도를 반환한다.
- 동일한 고정 스냅샷과 속도를 사용하면 `1000ms` 한 번과 `500ms + 500ms` 누적 이동량이 같아야 한다.
- 고정 속도가 플레이어를 통과할 수 있는 경우에도 도달한 시점의 전체 이동을 종료하며, 이미 목표에 도달한 분할 구간에서 반발 이동을 재개하지 않는다.
- 실제 이동 중 위치와 이웃 관계가 바뀌면 다음 업데이트에서 새 스냅샷으로 반발 방향을 다시 계산한다.
- 이 비선형 갱신 차이를 없애기 위한 fixed timestep은 이번 범위에 포함하지 않는다.

## 12. 장애물과 월드 경계

- 합성 속도로 계산한 `desiredEnd`를 기존 `moveCircleWithObstacles`에 전달한다.
- 군중 완화가 장애물 내부 또는 월드 밖 좌표를 직접 확정하지 않는다.
- 장애물에 막힌 반발 성분은 obstacle sweep 결과에 따라 줄어들 수 있다.
- 벽과 군중 사이에서 일부 겹침이 남는 것을 허용한다.
- 겹침을 없애기 위해 반복 projection, 다른 좀비 위치 강제 이동 또는 우회 경로 탐색을 추가하지 않는다.
- 군중 완화 때문에 좀비가 벽 앞에서 영구 정지하는지 수동 검증한다.

## 13. 결정론

- 공간 그리드가 반환한 canonical ID 순서를 사용한다.
- 반발 기여도는 위치 스냅샷에서 계산하고 동시에 적용한다.
- ID별 합산 결과는 입력 배열 순서와 무관해야 한다.
- 부동소수점 합산 순서는 canonical 후보 순서로 고정한다.
- 동일 중심 fallback은 ID 기반 결정론적 방향을 사용한다.
- 난수와 프레임 번호를 사용하지 않는다.

## 14. 성능 정책

- 후보 탐색은 기존 공간 그리드의 입력 4,096개 및 후보 15,000회 상한을 그대로 사용한다.
- 군중 계산은 완전한 후보 `k`에 대해 `O(k)` 시간과 좀비 수 `n`에 대해 `O(n)` 추가 메모리를 목표로 한다.
- 후보 전체가 불완전하면 `O(k)` 부분 계산을 위치에 적용하지 않는다.
- 반발 계산은 반복 수렴 루프를 사용하지 않는다.
- 프레임 간 상태, 누적 작업량 및 후보 캐시는 두지 않는다.
- 이번 작업에서 객체 풀, Web Worker 또는 추가 공간 인덱스를 만들지 않는다.

## 15. 책임 경계

### 공간 그리드

- 후보 ID 쌍, 유효성 및 완전성만 제공한다.
- 위치나 속도를 변경하지 않는다.

### 순수 군중 완화 로직

- 좀비 위치·반지름, 완전한 후보 쌍과 설정을 받아 ID별 반발 속도를 반환한다.
- Phaser, Scene, 장애물 및 플레이어를 참조하지 않는다.
- 피해, 체력, 공격 상태를 변경하지 않는다.

### `GameScene`

- 살아 있는 좀비 스냅샷을 만든다.
- 완전한 후보일 때만 순수 반발 결과를 기존 추적 이동에 합성한다.
- 최종 desired endpoint에 기존 장애물 이동을 적용한다.
- 접촉 경로와 플레이어–좀비 분리 순서를 유지한다.

## 16. 자동 테스트 요구사항

### 순수 반발

- 목표 최소 거리 밖의 쌍에는 반발이 없다.
- 목표 최소 거리 안의 쌍에는 크기가 같고 방향이 반대인 속도가 생긴다.
- 정확히 목표 최소 거리에서는 반발이 없다.
- 중심이 같은 쌍도 finite하고 결정론적인 반대 방향을 얻는다.
- 반지름이 0인 쌍은 NaN을 만들지 않는다.
- 여러 이웃의 기여도를 합산하고 최대 반발 속도를 넘지 않는다.
- 입력과 후보 순서를 바꿔도 ID별 결과가 같다.
- 중복 ID 또는 알 수 없는 후보 ID가 있으면 부분 결과 없이 실패한다.
- 비정상 설정과 좌표가 NaN 속도를 만들지 않는다.

### 이동 합성

- 후보가 없으면 기존 추적 방향과 속도가 유지된다.
- 반발이 있어도 최종 속도는 좀비 속도를 넘지 않는다.
- 최대 반발 속도가 추적 속도보다 작을 때 추적 진행 성분이 반대로 바뀌지 않는다.
- 고정 스냅샷에서 큰 delta와 분할 delta의 누적 이동량이 같다.
- 장애물과 월드 경계 적용 후 유효 위치를 유지한다.

### 공간 그리드 통합

- `valid: true`, `complete: true`일 때만 반발 결과를 사용한다.
- `valid: false`이면 모든 좀비가 기존 추적만 수행한다.
- `complete: false`이면 부분 후보를 적용하지 않는다.
- 입력 상한을 넘겨도 게임 루프가 중단되지 않고 추적 fallback을 사용한다.

### 기존 기능 회귀

- 접촉 피해는 군중 완화가 반영된 보정 전 이동 경로를 사용한다.
- 플레이어–좀비 충돌 보정 후 좌표가 접촉 endpoint를 덮어쓰지 않는다.
- 큰 delta와 분할 delta의 기존 접촉 피해 테스트를 삭제하거나 완화하지 않는다.
- 기존 플레이어–좀비 충돌 테스트를 삭제하거나 완화하지 않는다.
- 기존 공간 그리드 테스트를 삭제하거나 완화하지 않는다.
- `npm test`가 통과한다.
- `npm run build`가 통과한다.

## 17. 수동 검증 시나리오

- 30마리 이상이 한 방향에서 접근해도 모든 중심이 한 점에 장시간 쌓이지 않는지 확인
- 일부 몸 겹침이 남아 좀비 떼가 지나치게 정렬되지 않는지 확인
- 플레이어 추적 속도와 압박감이 기존과 크게 달라지지 않는지 확인
- 좁은 장애물 옆에서 좀비가 반발 때문에 영구 정지하지 않는지 확인
- 월드 모서리에서 위치가 유효한지 확인
- 접촉 중 공격 모션과 피해 시점이 기존과 일치하는지 확인
- 큰 delta를 유발하는 탭 전환 후 비정상적으로 튀지 않는지 확인
- 반복 restart 후 군중 관련 상태가 남지 않는지 확인

## 18. 완료 조건

- 공간 그리드의 완전한 후보만 군중 완화에 사용한다.
- 좀비가 목표 최소 거리 안에 있을 때 제한된 대칭 반발 속도를 얻는다.
- 좀비 일부 겹침을 허용하며 hard collision을 구현하지 않는다.
- 최종 속도가 기존 좀비 속도를 넘지 않는다.
- 반발이 플레이어 추적 진행을 완전히 뒤집지 않는다.
- 장애물과 월드 경계를 기존 로직으로 지킨다.
- 접촉 피해 경로와 플레이어–좀비 분리 계약을 유지한다.
- 입력 순서와 렌더 FPS가 같은 스냅샷의 반발 결과를 바꾸지 않는다.
- 불완전 후보를 부분 적용하지 않는다.
- 반복 수렴, 프레임 간 cursor 및 작업 backlog를 만들지 않는다.
- 기존 테스트를 삭제하거나 완화하지 않는다.
- 전체 테스트와 빌드가 통과한다.

## 19. 구현 후에도 의도적으로 남는 현상

- 좀비 몸 일부는 서로 겹칠 수 있다.
- 좁은 공간과 장애물 모서리에서는 겹침이 더 많이 남을 수 있다.
- 매우 큰 군중에서 후보 상한을 넘으면 해당 업데이트는 군중 완화 없이 기존 추적만 수행한다.
- 군중 완화는 길 찾기나 벽 우회를 해결하지 않는다.
- 좀비를 단단한 물리 개체처럼 밀어내지 않는다.

## 20. 현재 구현 상태

- `src/config/zombieCrowdSpacingConfig.ts`에 0.9 최소 거리 비율과 36px/s 최대 반발 속도를 정의했다.
- `src/logic/zombieCrowdSpacing.ts`에 후보 검증, 결정론적 대칭 반발, 합산 속도 제한, 속도 합성 및 delta 기반 이동을 구현했다.
- 공간 그리드 결과가 유효하고 완전할 때만 군중 반발을 계산한다.
- 불완전하거나 잘못된 후보 결과에서는 부분 반발 없이 기존 추적 이동을 사용한다.
- `GameScene`은 `zombieStarts` 스냅샷에서 반발을 계산하고 장애물 적용 전 추적 속도에 합성한다.
- 접촉 피해에는 군중 이동이 포함된 보정 전 `zombieMovementEnds`를 전달한다.
- 플레이어–좀비 충돌 보정과 이후 게임 상태 흐름은 변경하지 않았다.
