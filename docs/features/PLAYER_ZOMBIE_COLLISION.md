# 플레이어–좀비 충돌 방지 설계

## 1. 문서 상태

- 이 문서는 플레이어와 좀비 사이의 물리적 겹침 방지 요구사항을 정의한다.
- 현재 브랜치의 구현과 리뷰에서 확인된 시간·공간 경계를 함께 기록한다.
- 구현보다 이 문서의 상태 정의와 처리 순서를 우선한다.
- 좀비–좀비 충돌과 대규모 군중 처리는 이 문서의 범위가 아니다.

## 2. 작업 목적

- 플레이어와 좀비가 같은 공간에 겹쳐 보이지 않게 한다.
- 접촉 공격에 사용되는 실제 이동 경로를 충돌 보정 때문에 잃지 않는다.
- 큰 `deltaMs` 한 번과 같은 시간을 나눈 여러 업데이트에서 접촉 공격 결과가 달라지지 않게 한다.
- 장애물과 월드 경계 근처에서도 한 업데이트 안에 가능한 겹침을 해소한다.
- Phaser 객체와 무관하게 위치 계산과 접촉 경로를 테스트할 수 있게 한다.

## 3. 범위

### 포함

- 플레이어 원과 각 좀비 원 사이의 겹침 탐지
- 겹침 해소를 위한 플레이어 또는 좀비 위치 보정
- 장애물 및 월드 경계를 지키는 보정 이동
- 프레임 시작 위치, 이동 종료 위치, 충돌 보정 위치의 분리
- 충돌 보정 전 이동 경로를 이용한 접촉 시간 및 피해 판정
- 중심이 일치한 두 개체의 결정론적 분리 방향
- 물리적으로 해소할 수 없는 배치에서의 유한 종료

### 제외

- 좀비–좀비 충돌 및 분리
- 공간 그리드와 충돌 후보 최적화
- 프레임별 pair-check 예산과 프레임 간 작업 커서
- 좀비 경로 탐색과 군중 조향
- 장애물 생성 및 맵 구성 변경
- 접촉 피해량, 공격 간격, windup 및 무적 시간 밸런스 변경
- 플레이어·좀비 시각 효과 변경

## 4. 위치 상태 정의

한 업데이트에서 개체 위치는 다음 세 의미로 구분한다.

### `startPosition`

- 업데이트가 시작될 때의 위치다.
- 이전 업데이트에서 확정된 `resolvedPosition`과 같다.
- 이동 경로와 접촉 시간을 계산하는 시작점이다.

### `movementEnd`

- 입력, 추적 이동, 장애물 충돌 및 월드 경계를 적용한 직후 위치다.
- 플레이어–좀비 겹침 보정을 적용하기 전 위치다.
- 해당 업데이트 동안 개체가 이동한 물리 경로의 끝이다.
- 접촉 공격 판정은 반드시 `startPosition → movementEnd`를 사용한다.

### `resolvedPosition`

- `movementEnd`에서 플레이어–좀비 겹침을 해소한 최종 위치다.
- 렌더링, 카메라 추적 및 다음 업데이트의 `startPosition`에 사용한다.
- 이미 발생한 접촉 시간을 다시 계산하는 데 사용하지 않는다.

다음 대입은 금지한다.

```text
movementEnd = resolvedPosition
```

값이 우연히 같을 수는 있지만 두 상태의 의미와 보관 수명은 다르다.

## 5. 핵심 불변 조건

1. 접촉 판정 경로는 충돌 보정으로 짧아지거나 사라지지 않는다.
2. 최종 렌더링 위치는 가능한 경우 플레이어와 좀비의 반지름 합 이상 떨어져야 한다.
3. 충돌 보정 결과는 장애물 내부나 월드 경계 밖에 위치할 수 없다.
4. 같은 입력과 초기 상태에는 같은 결과를 반환해야 한다.
5. 중심이 일치해도 `NaN` 또는 무작위 방향을 만들지 않는다.
6. 보정은 한 업데이트 안에서 완료하며 다음 렌더 프레임을 정상 해결 수단으로 사용하지 않는다.
7. 물리적으로 분리할 공간이 없다면 무한 반복하지 않고 유한하게 종료한다.
8. 충돌 보정은 피해량, 공격 cooldown, windup 또는 무적 시간 규칙을 변경하지 않는다.

## 6. 처리 순서

`GameScene.update`의 관련 흐름은 다음 순서를 유지한다.

```text
1. playerStart와 zombieStarts 저장
2. 플레이어 이동 계산
3. 장애물·월드 경계를 적용한 playerMovementEnd 저장
4. 좀비 추적 이동 계산
5. 장애물·월드 경계를 적용한 zombieMovementEnds 저장
6. startPosition → movementEnd 경로로 접촉 window 계산
7. 기존 windup·cooldown·무적 시간 규칙으로 피해 처리
8. movementEnd를 입력으로 플레이어–좀비 겹침 해소
9. resolvedPosition을 Phaser 객체에 반영
10. 최종 위치를 기준으로 회전·카메라·HUD·시각 효과 갱신
11. 사망했다면 Game Over로 전환하고 이후 생성 처리를 하지 않음
```

접촉 판정을 8번 이후에 실행해야 한다면, Phaser 객체의 현재 좌표를 읽지 않고 별도로 보존한 `movementEnd`를 명시적으로 전달해야 한다. 처리 순서보다 데이터 의미 보존이 우선이다.

## 7. 접촉 공격 규칙

- 접촉 구간은 움직이는 두 원의 상대 이동으로 계산한다.
- 플레이어 경로는 `playerStart → playerMovementEnd`다.
- 각 좀비 경로는 같은 인덱스 또는 안정적인 ID로 연결된 `zombieStart → zombieMovementEnd`다.
- 충돌 보정으로 좀비가 플레이어 경계까지 밀려나더라도 보정 전 실제 겹침 시간은 유지한다.
- 최종 위치에서 두 원이 접하는 상태는 접촉으로 인정하는 기존 규칙을 유지한다.
- 공격 windup이 실제 접촉 구간 안에서 완료된 경우에만 피해가 발생한다.
- 큰 delta에서 발생한 접촉 시작 시각과 지속 시간은 delta 분할 결과와 일치해야 한다.

## 8. 겹침 해소 정책

- 각 개체는 설정된 원형 hit radius를 사용한다.
- 기본 우선순위는 플레이어 위치를 유지하고 좀비를 바깥으로 이동하는 것이다.
- 장애물이나 경계가 좀비 이동을 막아 겹침이 남으면 플레이어가 남은 거리만큼 반대 방향으로 이동할 수 있다.
- 보정 이동에도 기존 장애물 및 월드 경계 제약을 적용한다.
- 여러 좀비는 안정적인 ID 순서로 처리해 결과를 결정론적으로 유지한다.
- 플레이어 보정이 이미 처리한 좀비와 새 겹침을 만들면 같은 업데이트에서 다시 검사한다.
- 반복은 겹침 없음, 유효한 진행 없음 또는 보수적인 안전 상한에서 종료한다.
- 안전 상한은 성능 보호 장치이며, 상한까지 정상적으로 겹침을 남기는 것을 허용하는 완료 조건이 아니다.

### 중심이 일치하는 경우

분리 방향은 다음 순서로 선택한다.

1. 현재 두 중심 사이의 단위 방향
2. 두 개체의 이전 상대 위치에서 계산한 단위 방향
3. 고정되고 결정론적인 fallback 방향

렌더 프레임 번호, 난수 또는 현재 시각을 사용하지 않는다.

### 물리적으로 해소할 수 없는 경우

- 장애물과 경계 때문에 두 원을 배치할 유효 공간 자체가 없다면 무한 반복하지 않는다.
- 함수는 가장 겹침이 적고 유효한 위치를 반환해야 한다.
- 이런 결과를 정상적인 완전 분리로 보고하지 않도록 해소 여부를 결과로 표현하는 방안을 구현 시 검토한다.
- MVP 맵과 생성 위치는 이러한 불가능 배치를 만들지 않아야 한다.

## 9. 책임 경계

### 순수 충돌 로직

- 위치, 이전 위치, 반지름, 장애물, 월드 크기만 소비한다.
- Phaser `GameObject`, Scene, 카메라 또는 입력 객체를 참조하지 않는다.
- 최종 플레이어 위치와 좀비 ID별 최종 위치를 반환한다.
- 피해와 공격 상태를 변경하지 않는다.

### `DamageSystem`

- 현재 Phaser 좌표를 접촉 경로의 진실의 원천으로 가정하지 않는다.
- 시작점과 `movementEnd`가 명확한 경로 데이터를 사용한다.
- 기존 `movingCircleContactWindow`와 `resolveContactDamage` 규칙을 유지한다.

### `GameScene`

- 세 위치 상태의 수명을 관리한다.
- 이동 endpoint를 보존한 상태에서 접촉 판정과 겹침 보정을 연결한다.
- `resolvedPosition`만 Phaser 객체와 카메라에 최종 반영한다.
- 좀비–좀비 충돌이나 범용 물리 프레임워크를 추가하지 않는다.

## 10. delta 및 프레임 독립성 정책

- `deltaMs` 단위는 밀리초다.
- 이동량, 접촉 window, cooldown, windup 및 무적 시간은 경과 시간으로 계산한다.
- 위치 보정은 렌더 프레임 횟수가 아니라 현재 업데이트에서 주어진 기하 상태를 해결한다.
- 정상적인 해결을 다음 프레임으로 이월하지 않는다.
- 고정 pair-check 예산이나 프레임 간 후보 커서는 이 기능에 도입하지 않는다.
- 다음 두 실행은 같은 피해·공격 상태와 허용 오차 내 같은 비겹침 상태를 만들어야 한다.

```text
update(1000ms)
update(500ms) + update(500ms)
```

## 11. 자동 테스트 요구사항

### 순수 겹침 해소

- 비접촉 상태의 위치를 변경하지 않는다.
- 일반 겹침에서 플레이어를 유지하고 좀비를 완전히 밀어낸다.
- 중심이 일치해도 이전 상대 위치에 따라 결정론적으로 분리한다.
- 장애물이 좀비를 막으면 플레이어 쪽 보정으로 완전 분리한다.
- 월드 모서리에 낀 한 쌍을 한 호출에서 가능한 만큼 완전히 분리한다.
- 입력 순서가 달라도 안정적인 ID 기준 결과가 같다.
- 불가능한 배치에서도 유한 시간 안에 종료한다.

### 접촉 피해 통합

- 충돌 보정 전 경로에서 발생한 접촉 시간이 보존된다.
- 큰 delta에 좀비가 플레이어 안으로 진입했다가 분리되어도 windup 진행이 사라지지 않는다.
- 같은 총 시간을 한 번 또는 여러 번 업데이트한 피해 결과가 같다.
- 접촉하지 않은 이동에는 피해와 windup이 발생하지 않는다.
- 경계와 장애물 근처에서도 접촉 시간과 최종 비겹침을 함께 만족한다.
- 체력 0의 사망 전환은 한 번만 발생한다.

### 회귀 검증

- 기존 이동, 장애물 충돌 및 접촉 피해 테스트를 삭제하거나 완화하지 않는다.
- `npm test`가 통과한다.
- `npm run build`가 통과한다.

## 12. 완료 조건

- 플레이어와 좀비의 가능한 겹침이 같은 업데이트 안에서 해소된다.
- 접촉 피해가 `startPosition → movementEnd`를 사용한다.
- 충돌 보정 후 좌표가 접촉 판정용 endpoint를 덮어쓰지 않는다.
- 큰 delta와 delta 분할의 공격 결과가 일치한다.
- 장애물과 경계에서 유효 위치를 유지한다.
- 처리 결과가 결정론적이며 무한 반복하지 않는다.
- 좀비–좀비 충돌 및 공간 그리드가 추가되지 않는다.
- 관련 자동 테스트와 기존 전체 테스트가 통과한다.
- 프로덕션 빌드가 성공한다.

## 13. 현재 구현 상태

현재 브랜치는 다음 경계를 구현한다.

- 플레이어–좀비 겹침을 Phaser 비의존 순수 로직으로 해소한다.
- 장애물·경계 제약과 결정론적 분리 방향을 적용한다.
- `GameScene`은 이동 직후 `playerMovementEnd`와 `zombieMovementEnds`를 보존한다.
- `DamageSystem.resolveZombieContacts`는 Phaser 객체의 보정 후 좌표가 아니라 명시적인 이동 경로를 사용한다.
- 접촉 피해를 계산한 뒤 `resolvedPosition`을 Phaser 객체, 카메라 및 다음 프레임에 반영한다.
- 큰 delta와 분할 delta의 접촉 피해·windup·cooldown 일치를 자동 테스트로 검증한다.

좀비–좀비 충돌, 공간 그리드 및 프레임별 충돌 작업 예산은 구현하지 않았다.
