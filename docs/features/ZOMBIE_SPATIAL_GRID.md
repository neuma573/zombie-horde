# 좀비–좀비 공간 그리드 후보 탐색 설계

## 1. 문서 상태

- 이 문서는 좀비–좀비 근접 후보 탐색을 위한 고정 크기 공간 그리드 요구사항을 정의한다.
- 현재 브랜치는 이 문서를 기준으로 후보 탐색 순수 로직과 테스트를 구현한다.
- 구현 범위는 후보 탐색 최적화까지만 포함한다.
- 좀비 위치 보정과 실제 좀비–좀비 충돌 반응은 별도 작업이다.

## 2. 작업 목적

- 모든 좀비 조합을 검사하는 전수 탐색을 가까운 공간의 좀비 쌍 탐색으로 대체할 수 있는 순수 로직을 마련한다.
- 같은 위치 스냅샷에는 항상 같은 후보 순서와 결과를 제공한다.
- 후보 누락 없이 실제로 접촉하거나 겹칠 가능성이 있는 모든 좀비 쌍을 포함한다.
- 밀집 셀의 최악 비용과 메모리 사용을 숨기지 않고 명시적으로 다룬다.
- 낮은 FPS나 큰 `deltaMs`가 후보 결과를 바꾸지 않게 한다.
- 맵 크기와 카메라 viewport에 종속되지 않게 한다.

## 3. 현재 구현 분석

- `GameScene`은 각 업데이트에서 모든 좀비를 순서대로 이동한다.
- 플레이어 접촉 피해는 보정 전 이동 경로를 사용한다.
- `separatePlayerFromZombies`는 플레이어와 각 좀비의 겹침만 처리한다.
- 현재 좀비–좀비 후보 탐색, 충돌 판정 및 위치 분리는 존재하지 않는다.
- `Zombie.id`는 생성 순서를 포함한 안정적인 문자열 ID다.
- 현재 기본 좀비 hit radius는 20px지만, 그 값을 후보 누락을 일으키는 고정 가정으로 사용하지 않는다.
- 맵은 현재 2,400×1,600이지만 공간 그리드는 이 크기를 상수로 사용하지 않는다.
- 무한 웨이브는 웨이브가 증가할수록 좀비 수가 늘어나므로 밀집 상황의 최악 비용을 고려해야 한다.

## 4. 범위

### 포함

- 월드 좌표를 고정 크기 셀 좌표로 변환
- 좀비 위치 스냅샷으로 셀 인덱스 구성
- 같은 셀과 필요한 인접 셀의 후보 탐색
- 중복 없는 좀비 ID 쌍 생성
- 안정적인 후보 순서
- 서로 다른 반지름에서도 후보 누락이 없는 인접 셀 범위 계산
- 스트리밍 가능한 후보 열거
- 후보 생성 작업 상한과 명시적인 완료·중단 상태
- Phaser 비의존 단위 테스트

### 제외

- 좀비–좀비 겹침 해소 및 위치 변경
- 플레이어–좀비 충돌 로직 변경
- 장애물 충돌 및 경로 탐색 변경
- 좀비 조향, 회피, flocking 및 군중 AI
- 접촉 피해, hitscan, 자동 조준 및 웨이브 규칙 변경
- 좀비 수 제한이나 웨이브 밸런스 변경
- fixed timestep 도입
- Web Worker, ECS, quadtree 및 범용 물리 엔진
- 렌더링과 디버그 셀 표시

## 5. 데이터 경계

공간 그리드는 Phaser 객체 대신 다음 의미의 읽기 전용 스냅샷을 소비한다.

```ts
interface ZombieSpatialEntry {
  id: string;
  position: { x: number; y: number };
  radius: number;
}
```

후보 쌍은 ID 기준의 정규화된 형태다.

```ts
interface ZombieCandidatePair {
  firstId: string;
  secondId: string;
}
```

항상 다음 조건을 만족한다.

```text
firstId < secondId
```

정확한 타입명과 파일 위치는 구현 시 현재 `src/logic` 관례에 맞추되, Scene이나 `Zombie` 인스턴스를 타입에 포함하지 않는다.

## 6. 셀 정책

- 셀 크기는 월드 픽셀 단위의 양수 설정값 하나로 관리한다.
- 초기 셀 크기는 64px로 한다.
- 현재 좀비 지름 40px보다 크므로 일반 분산 상황에서 같은 셀과 가까운 인접 셀만 탐색할 수 있다.
- 후보 완전성은 셀 크기가 좀비 지름보다 크다는 가정에 의존하지 않는다.
- 각 엔트리의 반지름과 입력 전체의 최대 반지름으로 필요한 인접 셀 범위를 계산한다.
- 셀 좌표는 다음처럼 월드 원점을 기준으로 계산한다.

```text
cellX = floor(position.x / cellSize)
cellY = floor(position.y / cellSize)
```

- 음수 좌표도 같은 수식으로 결정론적으로 처리한다.
- 맵 resize, 카메라 scroll 및 viewport 크기는 셀 키 계산에 관여하지 않는다.
- 셀 키는 두 정수 좌표의 충돌 없는 표현을 사용한다.

## 7. 후보 완전성

- 두 원이 접하거나 겹칠 수 있다면 해당 두 좀비는 반드시 후보에 포함한다.
- 공간 그리드는 정확한 원–원 충돌 판정을 대신하지 않는다.
- 후보에는 실제로 멀리 떨어진 false positive가 포함될 수 있다.
- 후속 좁은 단계가 중심 거리와 반지름 합으로 실제 접촉 여부를 판정한다.
- 동일한 ID 쌍을 둘 이상의 셀 탐색에서 발견해도 한 번만 반환한다.
- 자기 자신과의 쌍은 반환하지 않는다.
- 유효하지 않은 반지름은 0으로 정규화한다.
- ID는 비어 있지 않고 입력 안에서 유일해야 한다.
- ID 중복, `NaN` 또는 무한대 좌표가 하나라도 있으면 전체 쿼리를 실패 처리한다.
- 잘못된 엔트리만 조용히 제외하거나 임의 셀에 넣지 않는다.
- 입력 실패 결과는 `valid: false`, `complete: false`, 빈 후보 목록과 오류 사유를 반환한다.

## 8. 결정론 정책

후보 결과는 입력 배열 순서와 Map의 삽입 순서에 의존하지 않는다.

1. 입력 엔트리를 `id` 오름차순으로 정규화한다.
2. 각 셀의 엔트리도 같은 정렬 순서를 유지한다.
3. 셀 탐색 순서는 `y` 오름차순 후 `x` 오름차순으로 고정한다.
4. 각 쌍은 작은 ID를 `firstId`로 정규화한다.
5. 최종 후보 순서는 `(firstId, secondId)` 사전순으로 정의한다.

같은 스냅샷과 설정에는 다음 값이 모두 같아야 한다.

- 후보 쌍 집합
- 후보 쌍 순서
- 검사한 후보 수
- 완료 또는 중단 상태

난수, 렌더 프레임 번호, 현재 시각 및 객체 메모리 순서를 사용하지 않는다.

## 9. 최대 비용 정책

### 복잡도 계약

- 그리드 구성과 결정론적 정렬은 좀비 수 `n`에 대해 `O(n)` 공간과 `O(n log n)` 시간이다.
- 후보 열거는 `O(k log c)`이며 `k`는 열거한 근접 후보 수, `c`는 해당 개체 주변의 점유 셀 목록 수다.
- 모든 좀비가 같은 셀에 몰리면 `k = n(n-1)/2`이므로 최악 시간은 `O(n²)`다.
- 공간 그리드는 밀집 군중의 수학적인 쌍 개수를 없애지 못한다.

### 메모리 및 작업 상한

- 후보 전체 배열을 먼저 `O(n²)` 크기로 생성하지 않는다.
- 후보는 iterator 또는 동등한 스트리밍 방식으로 한 쌍씩 열거한다.
- 인접 셀 occupant 목록은 최소 힙으로 병합하며, 후보마다 모든 목록을 다시 선형 탐색하지 않는다.
- 호출자는 양의 정수 `maxEntries`를 명시할 수 있다.
- 입력 수가 `maxEntries`를 넘으면 그리드를 구성하기 전에 `valid: false`, `complete: false`로 실패한다.
- 호출자는 양의 정수 `maxCandidateChecks`를 명시할 수 있다.
- 구현은 상한을 초과해 후보를 열거하거나 저장하지 않는다.
- 반환 결과는 최소한 다음 상태를 구분한다.

```ts
interface CandidateQueryStatus {
  checks: number;
  valid: boolean;
  complete: boolean;
}
```

- 한 번의 `check`는 실제 고유 canonical 후보 쌍 하나를 열거한 것을 뜻한다.
- 셀 구성, ID 정렬, 빈 인접 셀 방문 및 후보 개수 사전 계산은 `checks`를 소비하지 않는다.
- 후보 수를 알아내기 위한 별도 전체 사전 순회를 수행하지 않는다.
- 예산이 1이면 실제 후보 쌍을 최대 한 개 반환할 수 있어야 한다.
- 예산이 2이면 실제 후보 쌍을 최대 두 개 반환할 수 있어야 한다.
- `complete: true`는 결정론적 순회가 끝까지 도달했을 때만 반환한다.
- `complete: false`는 후보 집합이 불완전하다는 뜻이다.
- 불완전한 결과를 실제 충돌 반응이나 위치 보정의 authoritative 입력으로 사용하지 않는다.
- 상한 도달을 정상적인 “충돌 없음”으로 해석하지 않는다.
- 이번 후보 탐색 작업에서는 프레임 간 cursor나 미처리 후보 backlog를 만들지 않는다.

### 초기 상한

- 기본 입력 상한은 4,096개다.
- 구현 전용 기본 상한은 15,000회로 시작한다.
- 두 상한은 게임 밸런스가 아니라 전처리와 후보 폭발로 인한 브라우저 정지를 방지하는 한계다.
- 테스트는 상한 이하 실행과 `complete: false` 보고를 검증한다.
- 실제 좀비–좀비 충돌 반응을 연결하기 전, 현재 웨이브 규모로 complete 결과가 유지되는지 별도 측정한다.

## 10. 저 FPS 및 delta 정책

- 후보 탐색 입력은 한 시점의 위치 스냅샷이며 `deltaMs`를 받지 않는다.
- 같은 스냅샷의 후보 결과는 10 FPS, 30 FPS 및 60 FPS에서 동일하다.
- 큰 delta를 이유로 후보 검사 예산을 늘리거나 줄이지 않는다.
- 경과 시간에 비례한 작업 credit을 만들지 않는다.
- 남은 작업을 다음 렌더 프레임에 넘기는 cursor를 만들지 않는다.
- 따라서 후보 탐색 자체는 렌더 프레임 횟수로 진행률이 달라지지 않는다.
- `complete: false`이면 해당 결과를 부분 적용하지 않고 명시적으로 상위 계층에 보고한다.
- 향후 실제 충돌 반응이 불완전 후보를 어떻게 처리할지는 별도 설계에서 결정한다.
- fixed timestep이나 delta 기반 군중 해소를 이 작업에서 선행 구현하지 않는다.

## 11. 수명 주기와 갱신 정책

- 공간 그리드는 쿼리 호출마다 전달받은 스냅샷으로 새로 구성한다.
- Scene restart, Game Over 또는 resize 사이에 내부 위치 상태를 보존하지 않는다.
- 죽거나 제거된 좀비는 입력 스냅샷에 포함하지 않는다.
- 새로 생성된 좀비는 다음 쿼리 입력에 포함되면 즉시 인덱싱된다.
- 맵 크기가 바뀌어도 월드 좌표와 셀 크기만으로 다시 구성한다.
- 캐시 무효화, 객체 풀 및 증분 셀 갱신은 이번 작업에서 구현하지 않는다.

## 12. 책임 경계

### 순수 공간 그리드 로직

- 셀 구성과 후보 쌍 열거만 담당한다.
- 개체 위치, 체력, 공격 상태 및 Phaser 객체를 변경하지 않는다.
- 실제 원–원 겹침 해소를 수행하지 않는다.

### `GameScene`

- 이번 후보 탐색 구현 단계에서는 공간 그리드를 게임 루프에 연결하지 않는다.
- 후속 좀비–좀비 충돌 작업에서 살아 있는 좀비 위치 스냅샷을 제공할 수 있다.
- 후보 결과가 불완전할 때 임의로 일부 충돌만 적용하지 않는다.

### 기존 플레이어–좀비 충돌

- `separatePlayerFromZombies`의 입력, 처리 순서 및 접촉 경로 계약을 변경하지 않는다.
- 공간 그리드가 플레이어–좀비 후보를 대신 생성하지 않는다.

## 13. 자동 테스트 요구사항

### 셀 계산

- 같은 셀의 좀비가 후보가 된다.
- 셀 경계를 사이에 둔 접촉 가능 좀비가 후보가 된다.
- 필요한 인접 범위 밖의 좀비는 후보가 되지 않는다.
- 음수 좌표가 결정론적인 셀에 배치된다.
- 서로 다른 반지름에서도 접촉 가능 쌍이 누락되지 않는다.

### 중복과 결정론

- 자기 자신과의 쌍을 반환하지 않는다.
- 같은 ID 쌍을 중복 반환하지 않는다.
- 입력 순서를 뒤집어도 후보 목록과 순서가 같다.
- 동일한 위치의 여러 좀비도 ID 기준으로 안정적인 결과를 만든다.
- 반복 실행 결과가 완전히 같다.

### 비용 상한

- 분산된 대규모 입력에서 전수 조합보다 적은 후보만 검사한다.
- 한 셀에 밀집한 입력에서도 `maxCandidateChecks`를 초과하지 않는다.
- 상한 도달 시 `complete: false`를 반환한다.
- 불완전 상태를 `complete: true` 또는 후보 없음으로 잘못 보고하지 않는다.
- 후보 전체를 사전 배열로 생성하지 않아 상한 이전에 `O(n²)` 메모리를 할당하지 않는다.

### 회귀 검증

- 플레이어–좀비 충돌 테스트를 삭제하거나 완화하지 않는다.
- 접촉 피해의 큰 delta 및 분할 delta 테스트가 계속 통과한다.
- `npm test`가 통과한다.
- `npm run build`가 통과한다.

## 14. 완료 조건

- Phaser 비의존 공간 그리드 후보 쿼리가 존재한다.
- 현재 맵 크기나 viewport에 종속되지 않는다.
- 접촉 가능 좀비 쌍을 누락하지 않는다.
- 후보가 중복되지 않고 안정적인 ID 순서로 반환된다.
- 입력 순서와 FPS가 결과를 바꾸지 않는다.
- 후보 생성은 스트리밍 방식이며 명시한 작업 상한을 넘지 않는다.
- 불완전 결과는 `complete: false`로 구분된다.
- 불완전 후보를 게임 충돌 판정에 부분 적용하지 않는다.
- 좀비 위치 보정이나 좀비–좀비 충돌 반응을 구현하지 않는다.
- 기존 플레이어–좀비 충돌 및 접촉 피해 동작을 변경하지 않는다.
- 전체 테스트와 빌드가 통과한다.

## 15. 후속 작업 전 결정 사항

실제 좀비–좀비 충돌 반응을 연결하기 전에 다음을 별도 문서에서 결정한다.

- `complete: false`가 발생한 프레임의 게임플레이 정책
- 완전 분리, soft separation 또는 조향 중 어떤 반응을 사용할지
- 충돌 반응의 fixed timestep 필요 여부
- 밀집 군중에서 허용할 최소 거리와 시각적 겹침
- 장애물과 월드 경계에 막힌 군중의 해소 정책

이 결정이 끝나기 전에는 후보 쿼리를 실제 위치 보정에 연결하지 않는다.

## 16. 현재 구현 상태

- `src/logic/zombieSpatialGrid.ts`에 Phaser 비의존 후보 쿼리를 구현했다.
- 기본 셀 크기 64px와 기본 후보 검사 상한 15,000회를 적용한다.
- 기본 입력 상한 4,096개를 적용해 정렬과 그리드 전처리 비용도 제한한다.
- 입력 검증, 반지름 정규화, canonical ID 쌍 및 결정론적 순서를 구현했다.
- 후보는 전체 조합을 사전 생성하지 않고 ID별 인접 셀 목록을 최소 힙으로 병합해 점진적으로 수집한다.
- 상한 도달 시 `valid: true`, `complete: false`와 실제 처리한 후보만 반환한다.
- `src/tests/zombieSpatialGrid.test.ts`에서 셀 경계, 서로 다른 반지름, 결정론, 입력 오류, 1·2회 예산, 정확한 완료 및 밀집 입력 상한을 검증한다.
- `GameScene`에는 연결하지 않았으며 좀비 위치와 게임 판정은 변경하지 않는다.
